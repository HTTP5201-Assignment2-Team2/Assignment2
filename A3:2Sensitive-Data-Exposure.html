Introduction

To address security risks issue, OWASP has come up with a documents which represents a broad consensus about the most critical security risks to web applications. According to OWASR, companies should use this document as a guide to start changing the software development culture within the organization into one that produces more secure code. In terms of the risk assessment, OWASP TOP 10 analyses the following:
- Thereat agents/ Attack Vectors
- Security Weakness
- Impact
Based on the findings, the document provides an assessment of application vulnerability and recommends the ways to prevent an attack.

Sensitive Data Exposure

Many web applications and APIs do not properly protect sensitive data, such as financial, healthcare, and PII. Attackers may steal or modify such weakly protected data to conduct credit card fraud, identity theft, or other crimes. Sensitive data may be compromised without extra protection, such as encryption at rest or in transit, and requires special precautions when exchanged with the browser.

Thereat agents/ Attack Vectors

A manual attack is genearlly required. Attackets steal keys or clear text data of the server while in transit or form the user's client such as browser.

Security Weakness

The most common flaw is simply not encrypting sensitive data. When crypto is employed, weak key generation and management, and weak algorithm, protocol and cipher usage is common, particularly for weak password hashing storage techniques. For data in transit, server-side weaknesses are mainly easy to detect, but hard for data at rest

Impacts

Failure to encrypt data will compromise the sensitive personal information (PII) such as health records, credentials, personal data, and credit cards, which often require protection as defined by laws or regulations such as the EU GDPR or local privacy laws.

Is the application vulnerable?

For data in transit, server-side weaknesses are mainly easy to detect, but hard for data at rest. Therefore the first thing is to determine the protection needs of data in transit and at rest. PII require extra protection. For all such data:
- Is any data transmitted in clear text? This concerns protocols such as HTTP, SMTP, and FTP. External internet traffic is especially dangerous. Verify all internal traffic e.g. between load balancers, web servers, or back-end systems.
- Are any old or weak cryptographic algorithms used either by default or in older code?
- Are default crypto keys in use, weak crypto keys generated or re-used, or is proper key management or rotation missing?
- Is encryption not enforced, e.g. are any user agent (browser) security directives or headers missing?
- Does the user agent (e.g. app, mail client) not verify if the received server certificate is valid?

How to Prevent an attack

There are many ways to prevent an attack. They include but not limited to: - Classify data processed, stored or transmitted by an application. Identify which data is sensitive according to privacy laws, regulatory requirements, or business needs.
- Apply controls as per the classification.
- Donâ€™t store sensitive data unnecessarily. Discard it as soon as possible or use PCI DSS compliant tokenization or even truncation. Data that is not retained cannot be stolen.
- Make sure to encrypt all sensitive data at rest.
- Ensure up-to-date and strong standard algorithms, protocols, and keys are in place; use proper key management.
- Encrypt all data in transit with secure protocols such as TLS with perfect forward secrecy (PFS) ciphers, cipher prioritization by the server, and secure parameters. Enforce encryption using directives like HTTP Strict Transport Security (HSTS).
- Disable caching for response that contain sensitive data.
- Store passwords using strong adaptive and salted hashing functions with a work factor (delay factor), such as Argon2, scrypt, bcrypt or PBKDF2.
- Verify independently the effectiveness of configuration and settings.
